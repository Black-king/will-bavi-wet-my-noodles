# AGENTS.md

## Project

项目名称暂定为「台风巴威：杭州生存指南」。

这是一个面向普通用户的趣味互动网页，用更轻松、可分享、可操作的方式展示当前台风「巴威」对杭州的影响。项目不是传统气象业务系统，而是一个将实时台风路径、杭州本地风险提示、互动玩法和防台实用信息结合起来的静态网页。

## Current Phase

当前处于需求梳理阶段。

本阶段目标是先明确产品定位、数据策略、MVP 功能范围、视觉语气和部署方式，再进入设计与实现。除非用户明确要求，不急于写业务代码。

## User Intent

用户希望解决传统台风路径图的痛点：

- 页面严肃、难懂，普通人不愿意点开。
- 信息堆砌，用户需要自己判断台风离杭州多远、风险多高。
- 缺少互动和分享点。
- 数据接口是否稳定、是否收费、是否适合 GitHub Pages 部署存在不确定性。

用户当前关注的是 2026 年正在发生/发展的「巴威」台风，而不是单纯复盘历史台风。由于台风名称会循环使用，需求中必须避免把当前巴威误识别为 2020 年历史巴威。

## Product Direction

产品应以「杭州视角」为核心，而不是泛泛展示整个西北太平洋台风图。

优先回答用户最关心的几个问题：

- 巴威现在在哪里？
- 离杭州还有多远？
- 预计什么时候影响杭州附近？
- 杭州现在的风险等级是什么？
- 我需要做什么准备？

页面可以有趣，但不能削弱风险提示。所有娱乐化表达都应服务于理解，不应制造误导或过度恐慌。

## MVP Scope

首版 MVP 建议包含：

- Leaflet 地图，默认中心定位杭州及周边。
- 当前台风路径：实况路径与预报路径区分展示。
- 当前台风状态卡片：名称、编号、等级、中心位置、最大风速、中心气压、更新时间。
- 杭州距离与风险摘要：展示台风中心到杭州的距离、预计影响趋势和「杭州抗台指数」。
- 时间轴播放器：用户可以拖动查看历史路径点。
- 风圈可视化：展示 7 级、10 级、12 级风圈或在数据缺失时明确隐藏。
- 防台物资清单：手电筒、充电宝、饮用水、常用药、零食、窗户检查等。
- 数据来源与免责声明：注明来源、最后更新时间、预报存在不确定性。

首版暂不强求：

- 用户登录。
- 真实投票后端。
- 复杂海报生成后端。
- 多城市完整预警系统。
- 高精度灾害预测。

## Fun Layer

页面的趣味化可以围绕以下方向设计：

- 将「巴威」设计成一个有表情变化的台风 IP，风力越强表情越紧张。
- 使用杭州本地化文案，例如「西湖边适不适合散步」「今天外卖配送难度」。
- 用「生存指数」「宅家指数」「囤粮提醒」替代一部分冷冰冰的气象术语。
- 时间轴像视频播放器一样可拖动，让用户看到台风如何移动。
- 物资清单勾选后可以生成可复制的提醒文案。

趣味表达必须附带清晰数据依据，例如距离、风速、预报时间、数据更新时间。

## Data Strategy

推荐数据架构：

1. 前端只读取本仓库的静态 JSON，例如 `data/typhoon-bavi.json`。
2. GitHub Actions 定时调用第三方或官方数据源，把结果转换成统一 JSON。
3. GitHub Pages 部署静态页面，不在浏览器前端暴露 API Key。
4. 如果实时更新失败，页面继续使用上一次成功缓存，并显示数据延迟提示。

候选数据源：

- QWeather：优先候选，提供台风列表、实况路径、预报路径等接口。需要确认套餐、价格、调用限制和认证方式。
- 中央气象台台风网：权威参考源，适合人工核验或合规确认后的备用抓取，不应直接假设其页面接口长期稳定。
- 用户提供的 iStrongCloud 页面：适合作为交互参考和潜在数据线索，但是否可复用接口需要进一步确认。
- IBTrACS：适合台风结束后的历史复盘和对比，不适合作为当前实时主源。

当前数据层的关键要求：

- 不能把历史巴威和当前巴威混淆。
- 应通过年份、台风编号、英文名/中文名、活跃状态等字段识别当前台风。
- 前端数据结构要统一，避免 UI 直接依赖某一家 API 的原始字段。

## Deployment

首选 GitHub Pages。

推荐结构：

```text
/
├── AGENTS.md
├── index.html
├── style.css
├── script.js
├── data/
│   └── typhoon-bavi.json
├── img/
│   └── ...
└── .github/
    └── workflows/
        └── update-typhoon-data.yml
```

部署原则：

- 页面必须能在纯静态环境运行。
- 即使没有实时 API Key，也应能用样例 JSON 展示完整体验。
- 移动端优先，桌面端增强。

## Safety And Trust

这是天气灾害相关页面，必须保留基本严肃性：

- 明确写出数据来源。
- 明确写出最后更新时间。
- 明确说明页面仅供信息展示和生活提醒，正式预警以气象部门发布为准。
- 不用夸张标题制造恐慌。
- 在数据缺失、接口失败、更新时间过旧时给出明显提示。

## Open Questions

需求阶段需要继续确认：

- QWeather 的台风 API 当前套餐、价格、免费额度和调用限制。
- 是否要申请 QWeather Key，还是先用 mock JSON 做 MVP。
- 目标上线时间：是当天快速做出可分享版，还是做成完整作品。
- 页面语气：更偏可爱幽默、游戏化，还是偏杭州本地生活服务。
- 是否需要生成分享海报，若需要，首版用前端 Canvas 生成还是先做可复制文案。
- 是否需要接入用户定位，还是固定以杭州中心点计算距离。
- iStrongCloud 页面是否仅作参考，还是继续进行接口可用性调研。

## Requirement Workflow

需求梳理阶段按以下顺序推进：

1. 明确目标用户和使用场景。
2. 明确数据源与更新策略。
3. 明确 MVP 功能范围。
4. 明确视觉风格和交互语气。
5. 明确 GitHub Pages 部署约束。
6. 写出正式需求规格文档。
7. 用户确认后再进入实现计划。

## Definition Of Ready

进入实现前，至少需要满足：

- MVP 功能列表已确认。
- 数据源主方案和降级方案已确认。
- 是否使用真实 API Key 已确认。
- 页面核心文案风格已确认。
- 风险提示与免责声明已确认。
- 移动端优先的布局方向已确认。
